About marapendi
---------------
**marapendi** is a Python framework for physics-based modelling of proton-exchange
membrane (PEM) and anion-exchange membrane (AEM) electrochemical cells. 

*The current release targets PEM fuel cells. PEM/AEM electrolyzer support is planned for a future version.*

**marapendi** offers:

- **A basis for building fuel cell and electrolyzer models:**
  marapendi provides pre-defined correlations and sub-models for heat
  transfer, reaction kinetics, two-phase transport, membrane water balance,
  ohmic losses, and more (see :doc:`science/index`), with a focus on 0D
  PEM/AEM models. Sub-models are ordinary Python classes, so overriding a
  single correlation or swapping a whole sub-model is a matter of
  subclassing (see :doc:`user_guide/extending_models`).

- **Fast, practical calibration, validation and simulation:**
  marapendi brings ready-to-use, fast steady-state and transient 0D cell
  models. Their low computational cost makes sensitivity analysis, parameter
  estimation, cross-validation practical (see :doc:`science/parameter_estimation`). 
  Through an easy-to-use API, defining, calibrating and simulating cell models takes only 
  a few lines of code (see :doc:`installation` for a runnable example).

- **Open, readable and well-documented code:**
  Multiple runnable examples demonstrate marapendi's capabilities (see
  :doc:`auto_examples/index`), backed by detailed documentation and openly
  readable, commented code designed for transparency and easy understanding.

- **Cross-platform support:**
  A transient 0D model is available as a MATLAB/Simulink S-function block
  (see :doc:`user_guide/simulink_block`). Linking with the 
  `VirtualFCS <https://github.com/Virtual-FCS/VirtualFCS>`_ Modelica library is also
  planned for a future release.

**marapendi** is structured and written to make the implementation of models
and sub-models, and their use for simulation and parameter estimation, easy.
It separates the *description* of a cell (``components``) from the
*calculations* performed on it (``models``), and keeps the runtime *state* of
a simulation separate from both — every sub-model is an ordinary Python
class, meant to be subclassed rather than configured. See :doc:`architecture`
for the full breakdown of each subpackage.
